## Purpose

`docs/designs/` is the home for active PRISM design documents.

Use this directory for documents that define or explore:

- target architectures
- product-shape and capability families
- architecture companions to live contracts
- rollout-direction notes that are still shaping implementation
- validation and repo-configuration designs that are broader than one implementation slice

Design docs are not the same as contracts, specs, or roadmaps:

- contracts define stable seams and invariants
- specs define concrete implementation targets
- roadmaps define multi-phase execution order
- designs explain architecture shape, rationale, and intended direction before or alongside those more concrete docs

When a contract and a design doc overlap, the contract is normative.

## When to write a design doc

Write or update a design doc when the work includes one or more of:

- a target architecture that spans multiple subsystems
- a new product-surface or capability family that is still being shaped
- a policy or runtime model that needs review before it turns into a contract
- a repo-wide direction change that is too broad for a single spec
- a companion rationale document for a contract family

If the work is already implementation-ready, it probably belongs in `docs/specs/` instead.

## Filename rules

New design docs should use date-prefixed filenames:

- `YYYY-MM-DD-short-name.md`

Examples:

- `2026-04-09-warm-state-validation-feedback.md`
- `2026-04-09-repo-init-and-validation-ejection.md`

Migrated legacy design docs may keep their historical filenames until they are superseded or split.

## Recommended design structure

Design docs should usually include:

1. Summary
2. Goals
3. Non-goals
4. Core model or architecture
5. Lifecycle or state model where relevant
6. Query and UX expectations where relevant
7. Rollout guidance
8. Open questions or deferred follow-ons

Not every design doc needs every section, but a design doc should make the intended shape and boundaries easy to review.

## Relationship to other docs

Use this split:

- design = architecture shape, rationale, and target direction
- contract = normative seam or invariant
- spec = concrete implementation target
- roadmap = multi-phase sequencing
- ADR = durable accepted decision that later docs should cite

When a design matures:

- move stable normative rules into `docs/contracts/`
- create a spec in `docs/specs/` for the concrete delivery slice
- update or archive the design if it becomes historical only
